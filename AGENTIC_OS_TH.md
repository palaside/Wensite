# Agentic Skills Operating System (ระบบคำสั่งปฏิบัติการ)

ระบบคำสั่งและการลงสกิลต่อไปนี้คือแนวทางมาตรฐาน (Standard Protocols) สำหรับการทำงานร่วมกับ AI ในฐานะโปรเจกต์เมเนเจอร์ (Project Manager) เพื่อเน้นความโปร่งใส ปลอดภัย และป้องกันปัญหา Side Effect หรือ Context Drift ก่อนลงมือแตะต้องโค้ดบน Production

---

## 🛠️ โหมดการวิเคราะห์และออกแบบ (Analysis & Design Skills)

ในการเริ่มต้นโปรเจกต์ใหม่ หรือฟีเจอร์ใหม่ ให้ AI ทำงานผ่านระบบ Reverse Prompting โดยใช้คำสั่งต่อไปนี้:

### /analyze (โหมดนักวิเคราะห์และออกแบบสเปก)
- **การกระทำ:** สวมบทบาทเป็น System/Website Analysis Skill Generator 
- **ขั้นตอนการทำงาน:**
  1. ขอให้ผู้ใช้ส่ง Reference (รูปภาพ, Mockup, หรือ URL) และเป้าหมายเบื้องต้น
  2. เมื่อได้รับข้อมูล ให้ถามคำถามกลับ (Reverse Prompting) 1-3 ข้อเพื่อเจาะลึกรายละเอียด (เช่น Tone/Style, ฟีเจอร์หลัก, Framework ที่ต้องการ)
  3. สรุปความต้องการออกมาเป็น "สเปก" แบบฟอร์มตายตัว เพื่อให้ผู้ใช้อนุมัติก่อนไปต่อ
- **รูปแบบ Output สเปกที่บังคับใช้ (Template):**
  ```text
  /analyze
  # Objective: [เป้าหมายของงานแบบเข้าใจง่าย]
  # Natural Language Description: [คำอธิบายภาพรวม รูปแบบการใช้งาน และลูกเล่นต่างๆ]
  # Key Details:
    - Tone / Style: [เช่น หรูหรา, Minimal, Dark Mode]
    - Number of Pages / Sections: [จำนวนหน้า หรือ Section ที่ต้องมี]
    - Target Platform / Framework: [เช่น React, Next.js, Flutter, Python]
    - Primary Language: [เช่น TypeScript, Python, Go]
  # Example References: [URLs หรือชื่อรูปภาพที่ใช้เป็นต้นแบบ]
  # Adjustments: [ข้อกำหนดเพิ่มเติมพิเศษ]
  ```

## ⚙️ ระบบคำสั่งการทำงานหลัก (Workflow Commands)

เมื่อผ่านการอนุมัติสเปก (/analyze) แล้ว ให้ทำงานตามลำดับขั้นตอน (Phase) ดังต่อไปนี้อย่างเคร่งครัด:

### 1. /ASK (หรือ /GRILL-WITH-DOCS) — ขั้นเจาะลึกและตรวจสอบขอบเขต
- **การกระทำ:** ใช้เทคนิค Reverse Prompting ถามคำถามกลับ 1–3 ข้อ เพื่อเจาะลึกความต้องการ ขอบเขตของงาน และขจัดความคลุมเครือทั้งหมดก่อนเริ่มงานในแต่ละส่วนย่อย
- **กฎเหล็ก:** ห้ามเสนอโค้ดในขั้นตอนนี้เด็ดขาด ถามเพื่อเก็บข้อมูลและระบุขอบเขตให้ชัดเจนเท่านั้น

### 2. /PLAN (หรือ /UNDERSTAND-ANYTHING) — ขั้นวิเคราะห์โครงสร้างความเชื่อมโยง
- **การกระทำ:** นำข้อมูลมาวิเคราะห์โครงสร้างระบบในระดับภาพรวม บีบอัดบริบท (Context Compression) สกัดตัดทอนข้อมูลขยะออก 60%–95% เพื่อคงเหลือเฉพาะบริบทและลอจิกที่สำคัญ แล้วสร้างแผนงานอย่างเป็นระบบ (Systematic Plan)
- **กฎเหล็ก:** แสดงผลเป็น Checklist ทีละขั้นตอน (Step-by-step) ที่ชัดเจน ลำดับตามความจำเป็นก่อนหลัง (Topological Order) โดยระบุไฟล์ที่จะได้รับผลกระทบให้ครบถ้วน

### 3. /IMPLEMENT (หรือ /AGENTIC-EXECUTION) — ขั้นลงมือเขียนโค้ดและเรียกใช้เครื่องมือ
- **การกระทำ:** ลงมือเขียนโค้ดตามแผนในขั้นตอน PLAN อย่างเคร่งครัด
- **กฎเหล็ก:** ส่งมอบโค้ดแบบฉบับเต็มไฟล์ (Full Production-ready Code) ใน Code Block เดียว ห้ามเขียนย่อ ห้ามตัดทอน และห้ามอธิบายทฤษฎีแทรก

### 4. /REVIEW (หรือ /CAVEMAN) — ขั้นตรวจสอบและทำความสะอาดโค้ด
- **การกระทำ:** ตรวจสอบโค้ดที่ Implement ไป เปรียบเทียบความเปลี่ยนแปลง (Review Diff) เพื่อป้องกัน Side effect
- **กฎเหล็ก:** สรุปสั้นๆ ว่ามีจุดไหนถูกเพิ่ม ลบ หรือแก้ไขจากไฟล์เดิมบ้าง พร้อมบังคับใช้กฎลอกเปลือกนอกที่ไม่จำเป็นออกทั้งหมด (Anti-Slop) เน้นการออกแบบที่เรียบง่าย ปลอดภัย และทนทาน (Boring & Robust Design)

### 5. /TEST (หรือ /VERIFICATION-WORKFLOW) — ขั้นสอบทานความถูกต้อง
- **การกระทำ:** จำลองการรันระบบ หรือสร้างชุดทดสอบเพื่อพิสูจน์ว่าโค้ดทำงานได้จริง
- **กฎเหล็ก:** ให้คำสั่ง Command Line สำหรับการ Run/Build หรือเขียน Unit Test Code แบบสั้นๆ ที่ครอบคลุมเคสหลัก (Happy Path & Edge Case)

### 6. /COMMIT — ขั้นบันทึกสถานะระบบ
- **การกระทำ:** สรุปงานทั้งหมดเพื่อเตรียมบันทึกเข้าระบบ Version Control
- **กฎเหล็ก:** สร้าง Commit Message ที่ได้มาตรฐาน (เช่น Conventional Commits: `feat:`, `fix:`, `refactor:`) พร้อมคำอธิบายสั้นๆ 1 บรรทัด
